Use this workflow when requests stall, migrations wait, or the instance overview reports lock pressure.

## Before you start

- Give the managed-instance login `pg_monitor` so other sessions and queries are visible.
- Confirm the incident is still active. Querylane takes snapshots and does not retain a lock-event history.
- Know your organization's rule for canceling a query or terminating a backend.

Querylane explains blocking but does not cancel or terminate sessions.

## 1. Open the blocking chain

Open the instance, select **Activity**, and refresh.

When PostgreSQL reports blockers, Querylane places a blocking-chain card above the session table. Read it from the root blocker to the waiting sessions. A blocker can itself wait behind another backend, so follow the complete chain before choosing a PID.

If a blocker is not present in the sample, Querylane says so. The session may have ended, fallen outside the 50-session snapshot, or be hidden by the registered role's statistics visibility.

## 2. Inspect the root blocker

Select the root blocker and record:

- PID;
- role and database;
- application name and client address;
- session, transaction, and query ages;
- session state;
- full query text; and
- downstream sessions it blocks.

`idle in transaction` deserves special attention. The last query can be complete while its transaction still retains locks.

Do not assume the longest-running query is the blocker. Use the **Locks & waits** section and the explicit blocking relationship.

## 3. Inspect the impact

Open representative waiting sessions and compare:

- how long each has waited;
- which applications and databases are affected;
- the lock wait event;
- whether one blocker fans out to many sessions; and
- whether retries are adding more waiting work.

This separates one delayed maintenance task from a blocker on a critical request path.

## 4. Verify before intervention

Refresh Activity immediately before acting. PostgreSQL PIDs can disappear, and a new backend can eventually reuse a PID.

Confirm the same session through your approved PostgreSQL channel. For example:

```sql
SELECT
  pid,
  usename,
  application_name,
  client_addr,
  state,
  xact_start,
  query_start,
  wait_event_type,
  wait_event,
  query
FROM pg_stat_activity
WHERE pid = 1842;
```

Use the actual PID from Querylane. Run this as an appropriately authorized operations role, not by expanding Querylane's registered role solely for intervention.

## What to do next

Choose the least disruptive action outside Querylane:

1. Ask the owning application or operator to commit or roll back the transaction.
2. Cancel the active statement if ending the query is sufficient.
3. Terminate the backend only when canceling cannot release the transaction and your runbook permits it.
4. Watch the blocked sessions recover and refresh Querylane.

After the incident, fix the cause: transaction scope, missing timeout, migration coordination, connection-pool behavior, or application retry storms. Save the blocker context and affected applications in the incident record.
