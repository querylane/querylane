MetricsService serves the time series collected by querylane's background sampling probes, turned into chart-ready shape: samples are bucketed to a requested resolution, cumulative counters are converted to per-second rates, and an optional period-over-period trend delta is computed server-side. It is a read API over the meta database's sample tables; it never touches the managed instances. The live single-point equivalents live on InstanceService.GetInstanceOverview.

## QueryMetrics

QueryMetrics returns one time series per requested metric for a target (an instance or a database) over a time window.

```protobuf
rpc QueryMetrics(QueryMetricsRequest) returns (QueryMetricsResponse)
```

- **Procedure:** `POST /querylane.console.v1alpha1.MetricsService/QueryMetrics`
- **Mode:** Unary

### JSON request example

The values are illustrative. Replace resource names, identifiers, and credentials for your deployment.

```json
  },
  "step": "60s",
  "comparison": "3600s"
}
```

### Request

`QueryMetricsRequest`

| Field | Type | Behavior | Description |
| --- | --- | --- | --- |
| `target` | `string` | Required | Required. The resource whose metrics to query, as an AIP resource name: `instances/{instance}` for instance-scoped metrics or `instances/{instance}/databases/{database}` for database-scoped metrics. Each requested metric declares which scope it belongs to; a metric whose scope does not match the target is rejected. Fails with NOT_FOUND when the target's instance does not exist. |
| `metrics` | `repeated MetricId` | Required | Required. The metrics to return. Each yields one MetricSeries in the response (or one entry in partial_errors if it could not be served). |
| `interval` | `Interval` | Required | Required. The half-open window `[start_time, end_time)` to query (AIP-145). Relative ranges like "last 24h" are resolved to absolute timestamps by the client. |
| `step` | `Duration` | Optional | Optional. The requested bucket width (resolution). The server clamps it per metric to at least that metric's probe cadence and to a total-point ceiling; each series echoes the width it actually used in Points.step. When unset the server picks a width that yields a chart-friendly number of points. |
| `comparison` | `Duration` | Optional | Optional. A period offset for period-over-period comparison (for example, 24h, 7d). When set, each series carries a TrendDelta comparing the window against the same-length window shifted back by this offset. The previous-period line itself is not returned; a client that wants to overlay it issues a second QueryMetrics with the interval shifted back by the same offset. |

### Response

`QueryMetricsResponse`

| Field | Type | Behavior | Description |
| --- | --- | --- | --- |
| `interval` | `Interval` | — | The requested window, echoed back. A series' buckets cover the window and may extend slightly past end_time when its step does not divide the window evenly; the trailing partial bucket only aggregates samples before end_time. |
| `step` | `Duration` | — | The finest bucket width used across the returned series. Each series' Points.step is authoritative: a metric with a coarser probe cadence uses a coarser width. |
| `series` | `repeated MetricSeries` | — | One series per successfully-served metric, in request order. |
| `partialErrors` | `repeated Status` | — | Per-metric failures. The RPC still returns OK; a metric that could not be served (for example, an IO metric on PostgreSQL < 16, or a permissions gap) is omitted from `series` and reported here. The failing metric is identified by an ErrorInfo detail with metadata `{"metric": "<METRIC_ID_...>"}` — mirrors GetInstanceOverviewResponse.partial_errors. |
